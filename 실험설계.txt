## 1. Project Goal

본 프로젝트의 목표는 `LLaVA-OneVision-Qwen2-0.5B-SI`를 STM32N6570-DK에서 직접 실행하기 전에, 모델을 구성하는 주요 연산들의 latency와 memory 사용량을 정량적으로 분석하는 것이다.

최종적으로는 원본 LLaVA 0.5B를 그대로 MCU에 올리는 것이 아니라, NAS 또는 distillation을 통해 축소된 LLaVA-style VLM을 사용할 예정이다. 따라서 본 실험은 축소 모델을 설계하기 위한 사전 profiling 단계에 해당한다.

구체적으로, LLaVA forward path를 operator 단위로 분해하고, 각 operator를 TinyEngine 기반 microbenchmark로 구현하여 STM32N6570-DK에서 latency와 memory를 측정한다. 이를 통해 어떤 연산 또는 module이 MCU 환경에서 가장 큰 병목인지 확인하고, NAS/distillation 과정에서 어느 부분을 얼마나 줄여야 하는지 결정한다.

---

## 2. Motivation

MCU에서 LLaVA-style Vision-Language Model을 실행하려면 원본 모델을 그대로 사용하는 것은 어렵다. 특히 vision token 수, Qwen decoder의 hidden dimension, MLP intermediate dimension, layer 수, KV cache, lm_head 등이 latency와 memory 측면에서 큰 병목이 될 수 있다.

그러나 무작정 모델을 축소하면 정확도가 크게 떨어질 수 있다. 따라서 먼저 실제 target board인 STM32N6570-DK에서 각 연산이 어느 정도의 latency와 memory cost를 갖는지 측정해야 한다.

이 profiling 결과는 이후 NAS 또는 distillation 기반 모델 축소 과정에서 다음과 같은 결정을 내리는 근거로 사용된다.

```text
1. Vision encoder를 얼마나 줄일 것인가?
2. Image token 수를 몇 개까지 허용할 것인가?
3. Qwen decoder layer 수를 얼마나 줄일 것인가?
4. Hidden dimension을 어느 정도로 줄일 것인가?
5. MLP intermediate dimension을 얼마나 줄일 것인가?
6. Attention sequence length와 KV cache budget을 어떻게 제한할 것인가?
7. lm_head 또는 vocab projection을 어떻게 줄일 것인가?
8. 어떤 operator를 우선적으로 최적화하거나 대체할 것인가?

즉, 본 실험은 최종 모델을 바로 만드는 단계가 아니라, MCU-friendly LLaVA 구조를 설계하기 위한 hardware-aware profiling 단계이다.

3. Role of Profiling Results

본 실험에서 얻은 latency/memory 결과는 NAS 또는 distillation된 LLaVA 0.5B 모델의 축소 방향을 결정하는 데 사용된다.

예를 들어, 특정 operator가 latency 병목이라면 해당 module의 dimension이나 layer 수를 줄이는 방향을 고려한다. 특정 activation이나 KV cache가 memory 병목이라면 image token 수, sequence length, cache 구조를 제한하는 방향을 고려한다.

Profiling result
  ↓
Bottleneck analysis
  ↓
Compression target decision
  ↓
NAS / Distillation search space design
  ↓
MCU-friendly LLaVA model

따라서 본 프로젝트의 핵심 질문은 다음과 같다.

"STM32N6570-DK에서 LLaVA-style VLM을 실행하려면,
어떤 연산을 얼마나 줄여야 하는가?"

4. Profiling-guided Compression Plan

Profiling 결과는 다음과 같은 축소 계획에 활용된다.

Profiling Target	Measurement	Compression Decision
Vision projector	image token 수별 latency/memory	image token budget 결정
Qwen Linear	hidden size별 latency/memory	hidden dimension 축소 범위 결정
Qwen MLP	intermediate size별 latency/memory	MLP expansion ratio 축소 범위 결정
Attention	sequence length별 latency/memory	max sequence length / KV cache budget 결정
KV Cache	layer 수, sequence length별 memory	decoder layer 수 및 cache 구조 결정
lm_head	vocab size별 latency/memory	vocab 축소 또는 candidate head 필요성 판단
Mini Qwen Block	block-level latency/memory	전체 layer 수 예산 추정
Mini LLaVA Path	image-token + text-token path 측정	end-to-end MCU feasibility 추정

5. Expected Outcome

본 실험의 최종 결과는 단순한 operator benchmark가 아니라, NAS/distillation을 위한 hardware-aware design guideline이다.

최종적으로 다음과 같은 질문에 답하는 것을 목표로 한다.

1. STM32N6570-DK에서 감당 가능한 image token 수는 얼마인가?

2. Qwen hidden dimension 896은 너무 큰가?
   줄인다면 어느 정도까지 줄여야 하는가?

3. MLP intermediate dimension 4864는 latency 병목인가?
   NAS search space에서 어느 범위로 제한해야 하는가?

4. Decoder layer 24개는 현실적인가?
   layer 수를 줄여야 한다면 어느 정도가 적절한가?

5. KV cache는 SRAM 안에 유지 가능한가?
   sequence length 제한이 필요한가?

6. lm_head full vocab projection은 가능한가?
   candidate vocab 또는 task-specific head가 필요한가?

7. NAS/distillation된 LLaVA 0.5B를 설계할 때
   가장 먼저 줄여야 하는 module은 무엇인가?
   
6. Summary

본 프로젝트는 LLaVA-OneVision-Qwen2-0.5B-SI를 STM32N6570-DK에서 바로 실행하는 것을 목표로 하지 않는다.

대신 LLaVA의 주요 operator를 TinyEngine 기반으로 하나씩 구현하고, latency와 memory를 측정한다. 이 결과를 바탕으로 NAS 또는 distillation을 통해 축소된 LLaVA-style VLM을 설계할 때 어떤 부분을 얼마나 줄여야 하는지 결정한다.

즉, 본 실험은 MCU-friendly LLaVA 모델을 만들기 위한 hardware-aware profiling 단계이다.
