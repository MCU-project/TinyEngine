2. 원본 코드에서 확인된 문제
---------------------------

1) RoPE 의미론 오류
- 파일:
  TinyEngine/src/kernels/fp_backward_op/llm_ops_fp.c
- 문제:
  rotary_embedding_fp가 Qwen-style rotate_half가 아니라 인접 pair 회전 방식으로 구현되어 있었음.
- 영향:
  Qwen/LLaVA(Qwen 기반) torch 기준과 의미론이 다름.

2) element count uint16_t truncation
- 파일:
  TinyEngine/src/kernels/fp_backward_op/mm_projector_fp.c
  TinyEngine/src/kernels/fp_backward_op/qwen_block_fp.c
- 문제:
  큰 tensor 크기를 uint16_t로 넘겨 일부 element만 처리될 수 있었음.
- 영향:
  실제 모델 크기에서 결과가 크게 틀어질 수 있음.

3) GELU 정의 차이
- 파일:
  TinyEngine/src/kernels/fp_backward_op/llm_ops_fp.c
- 문제:
  gelu_fp는 torch default GELU가 아니라 tanh approximation GELU였음.
- 영향:
  torch.nn.functional.gelu() 기본값과 완전히 같지 않음.

6. 수정본에서 적용한 핵심 수정
------------------------------

1) Qwen-style RoPE 적용
- rotary_embedding_qwen_fp에서 rotate_half 의미론으로 변경.

2) elementwise truncation 제거
- qwen_block verified 내부 add/mul/silu를 size_t 기반 loop로 분리.
- mm_projector verified도 size_t 기반 element count 사용.

3) exact GELU 추가
- torch default GELU와 맞는 경로 추가.

4) fast GELU 분기 추가
- STM32용 경로에서 exact GELU 대신 tanh 근사 GELU를 size_t 기반으로 선택 가능하게 추가.

5) checked API 추가
- workspace 크기 검사 가능
- rope table row 수 검사 가능
- position_ids가 rope table 범위를 넘는지 검사 가능

6) attention score workspace 축소
- qwen_block verified에서 attention score 버퍼를 full seq_len * seq_len이 아니라
  1 row만 쓰는 streaming 방식으로 축소.

11. STM32N6570-DK 실사용 관점에서 파악한 취약점
----------------------------------------------

1) exact GELU의 MCU 비용
- gelu_exact_fp_verified는 erff() 사용
- 정확도 baseline으로는 좋지만 MCU 실사용 기본 경로로는 무거울 수 있음

2) workspace 총량 문제
- qwen_block verified는 attention score 버퍼는 줄였지만
  여전히 seq_len * hidden, seq_len * intermediate 버퍼가 큼
- 실제 큰 Qwen/LLaVA 크기에서는 여전히 메모리 부담이 큼

3) checked API 이전의 안전성 문제 보완
- workspace size 검사
- rope row count 검사
- position_ids 범위 검사
를 추가해 런타임 OOB 위험을 줄였음

4) 아직 남은 한계
- API config 필드 상당수가 uint16_t 기반
- 실모델 전체를 STM32N6570에서 곧바로 돌릴 수준의 메모리 최적화는 아직 아님
